## MODIFIED Requirements

### Requirement: 系統必須自動取得使用者的 Google 帳號
系統 MUST 透過前端的 Google Identity Services (GIS) 取得使用者的 ID Token (JWT)，並將其傳送至後端進行驗證與解析以取得 Google 帳號 Email，作為後續權限驗證的基礎。

#### Scenario: 使用者首次存取應用程式
- **WHEN** 使用者開啟網頁應用程式網址且尚未登入
- **THEN** 系統會顯示帶有 Google 登入按鈕的畫面，提示使用者授權並登入

#### Scenario: 前端傳送無效的 Token
- **WHEN** 前端傳送偽造或過期的 JWT 至後端
- **THEN** 後端驗證失敗並拒絕請求，回傳未授權錯誤

## ADDED Requirements

### Requirement: 系統必須根據使用者角色動態渲染前端介面 (SPA)
系統前端 MUST 在成功驗證身分並取得使用者資料後，根據使用者的「角色」屬性，動態隱藏或顯示對應的應用程式區塊（如學生介面、導師介面）。

#### Scenario: 學生登入顯示學生介面
- **WHEN** 驗證通過且使用者角色為「學生」
- **THEN** 前端隱藏登入區塊，顯示專屬學生的功能區塊，並載入相關資料

#### Scenario: 導師登入顯示導師介面
- **WHEN** 驗證通過且使用者角色為「導師」
- **THEN** 前端隱藏登入區塊，顯示專屬導師的功能區塊，並載入相關資料

### Requirement: 系統必須在後端執行嚴格的資料存取防護
後端 API MUST 在處理任何資料請求時，驗證傳入的 JWT 並確認其對應的「角色」是否具備存取該資料的權限。不可僅依賴前端隱藏機制。

#### Scenario: 越權請求資料被阻擋
- **WHEN** 使用者角色為「學生」，但嘗試透過前端 API 呼叫獲取「管理員」或「導師」專屬的資料（例如全校成績單）
- **THEN** 後端 API 檢查權限失敗，直接回傳拒絕存取錯誤，不回傳任何實際資料

### Requirement: 系統必須驗證使用者是否在白名單中
系統 MUST 比對由 JWT 解析出的 Email 是否存在於「帳號管理」試算表的啟用名單中。

#### Scenario: 授權使用者登入
- **WHEN** 解析出的 Email 存在於白名單且狀態為「啟用」
- **THEN** 後端回傳使用者資料（包含角色資訊），前端依此載入主要應用程式介面

#### Scenario: 未授權使用者登入
- **WHEN** 解析出的 Email 不存在於白名單或狀態不為「啟用」
- **THEN** 後端拒絕提供資料，前端顯示未授權引導頁面，並隱藏所有敏感資訊

### Requirement: 系統必須支援特定網域限制
系統 MUST 能夠根據「網站參數設定」中的「網站網域」設定，限制只有該網域的使用者才能存取。（注意：此功能需設計成可選，或可依角色設定，以支援跨網域的學生）

#### Scenario: 網域符合的使用者登入
- **WHEN** 由 JWT 解析出的 Email 網域符合系統設定的「網站網域」或系統未嚴格限制網域
- **THEN** 系統允許進行後續的白名單驗證流程

#### Scenario: 網域不符合的使用者登入
- **WHEN** 系統設定了全局網域限制，且由 JWT 解析出的 Email 網域不符合該設定
- **THEN** 後端直接拒絕存取，前端顯示未授權頁面
